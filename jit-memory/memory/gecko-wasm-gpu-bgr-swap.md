GPU mode (in-process WebRender -> WebGL2 -> page canvas; see [[gecko-wasm-gpu-integration]]) swapped R↔B on RASTER IMAGES only (`<img>`/decoded bitmaps): a solid-red PNG read back blue `[0,0,255,255]`. Solid fills, gradients, text, vertex colors were fine. **FIXED 2026-06-17** with a shader-side swizzle (CPU per-pixel swizzle rejected as too slow). Verified: embed-xul/gpu-bgr-test.cjs renders a red `<img>` RED, a green CSS div green, and a yellow->blue gradient correct (screenshot embed-xul/gpu-bgr.png).

ROOT CAUSE: Firefox image data is BGRA8. WebRender (`gfx/wr/webrender/src/device/gl.rs` Device::new) wants `Swizzle::Bgra` for BGRA8 images and normally applies it via GL_TEXTURE_SWIZZLE. WebGL2 removed core texture swizzle AND emscripten doesn't advertise GL_EXT_texture_format_BGRA8888, so it hit branch 4 (RGBA8, no swizzle) -> BGRA bytes uploaded as RGBA -> swapped. The deeper blocker for any fix: `Device::swizzle_settings()` returns `None` whenever `supports_texture_swizzle` is false, and `texture_cache.rs` only tags an item `Swizzle::Bgra` when swizzle settings are `Some` (`match self.swizzle { Some(_) => Bgra, None => default }`, ~L1350). So on emscripten the texture cache disabled swizzle ENTIRELY -> images got `Swizzle::default()` (Rgba), no signal to distinguish images from RGBA render-targets.

THE FIX (all in firefox/gfx/wr/webrender, needs gkrust rebuild + relink):
1. `src/device/gl.rs` Device::new branch 4 (the `gl::GlType::Gles =>` catch-all): on emscripten set `bgra8_sampling_swizzle = Swizzle::Bgra` (the swizzle VALUE for BGRA8 images).
2. `src/device/gl.rs` `swizzle_settings()`: return `Some(self.swizzle_settings)` when `supports_texture_swizzle || cfg!(target_os="emscripten")`. THIS is the key fix — it makes texture_cache tag BGRA8 images `Swizzle::Bgra` (and sets `use_upload_format=false` so uploads stay BGRA-in-RGBA, a valid WebGL2 RGBA8 upload). bind_texture_impl still skips the actual GL_TEXTURE_SWIZZLE (gated on supports_texture_swizzle=false) so no GL_INVALID_ENUM on WebGL2. Only BGRA8 images get Bgra; R8 glyph masks (format matches -> default) and RGBA render-targets stay unswizzled.
3. Plumb the swizzle to the shader as a per-bind uniform: `Program.u_color0_swizzle` (looked up "uColor0Swizzle" at link), `Device.bound_program_color0_swizzle_loc` (cached in `bind_program`), and `bind_texture` sets `uColor0Swizzle = (swizzle==Bgra?1:0)` for the Color0 slot (slot_index 0), gated `#[cfg(target_os="emscripten")]`. NOTE: capture `slot.0` into a local BEFORE `bind_texture_impl(slot,...)` — TextureSlot is NOT Copy, so using slot.0 after the move is a compile error.
4. `res/shared.glsl` (fragment block): `uniform int uColor0Swizzle;` + `vec4 swizzleColor0(vec4 c){ return uColor0Swizzle==1 ? c.bgra : c; }` (c.bgra swaps R<->B). No-op when 0, so inert on desktop (uniform never set there).
5. Wrap the sColor0 color sample with `swizzleColor0(...)` in: `res/brush_image.glsl`, `res/sample_color0.glsl` (ps_quad_textured), `res/ps_quad_repeat.glsl`. Safe to wrap any sColor0 sample because the uniform mirrors the actual per-bind swizzle (render-targets -> 0 -> no-op). NOT YET WRAPPED (still potentially swapped, follow-up): color glyphs (ps_text_run), YUV video (brush_yuv_image/yuv), compositor external images (composite.glsl).

BUILD GOTCHAS: shaders are embedded in the gkrust lib via webrender build.rs (`include_str!` unoptimized + glslopt-optimized); `strings libxul.so | grep uColor0Swizzle` confirms embedding (plain `grep` skips the binary -> false negative). `./mach build` deadlocks/half-fails here (see [[gecko-wasm-build-mach-deadlock]]) AND `mach build binaries` errors `No rule to make target 'wr/webrender/binaries'` and orphans a stuck cargo -> use `make -C obj-full-emscripten/toolkit/library -j$(nproc)` (rebuilds gkrust + relinks libxul), then embed-xul/restrip-relink-web.sh. Capture FULL build output to a file (don't `tail`-truncate) to see the real rustc error.

VERIFIED in embed-chrome too (2026-06-17): the red `<img>` renders red in the full-chrome GPU build. Getting embed-chrome's ?gpu=1 to even start needed two unrelated ports (embed-chrome's harness drifts from embed-xul because `make web` only sets up embed-xul; embed-chrome shares gecko.{wasm,data,js} via symlinks but index.html is hand-maintained): (1) add the `#gldummy` 1x1 canvas to embed-chrome/index.html (OFFSCREENCANVASES_TO_PTHREAD=#gldummy; without it startup aborts `could not find canvas "#gldummy"`); the engine auto-creates #glout itself (EM_ASM in gecko.js) and the z-index layering works without #screenwrap. (2) stage fonts at `/gre/browser/fonts` — the chrome build's binDir is /gre/browser (GRE/APP split) so gfxFT2FontList::FindFonts scans `<NS_XPCOM_CURRENT_PROCESS_DIR>/fonts` = /gre/browser/fonts, NOT /gre/fonts; empty -> MOZ_CRASH("No font files found") -> unreachable. Fixed in embed-xul/stage-gre.sh (stages BOTH /gre/fonts and /gre/browser/fonts). See [[gecko-wasm-full-chrome]]. Test: embed-chrome/gpu-chrome-bgr-test.cjs.
