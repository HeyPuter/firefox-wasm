`embed-chrome/` renders the COMPLETE Firefox front-end (browser.xhtml: tabs, URL bar, toolbar) headless in wasm and loads real sites in tabs (example.com over WISP verified, screenshot embed-chrome/site-in-chrome.png). Shares gecko.wasm/.data with embed-xul (symlinked); selected at runtime by env `GECKO_CHROME=1`. All logic lives in `embed-xul/embed-xul.cpp`, gated on `getenv("GECKO_CHROME")`. See [[gecko-wasm-runtime-frontier]] and [[gecko-wasm-source-patches]].

The non-obvious bring-up sequence (each was a distinct blocker, found by symbolizing/probing — see [[gecko-wasm-toolchain]] for DWARF symbolization, NOT printf bisection):
- Use `nsIAppShellService::CreateTopLevelWindow(browser.xhtml, CHROME_ALL)`, NOT `CreateWindowlessBrowser` — the front-end installs XULBrowserWindow on an nsIAppWindow tree owner; windowless has none so gBrowser never inits.
- `setenv("MOZ_HEADLESS","1")` before gfx init — AppWindow only calls `CreateHeadlessWidget()` when `gfxPlatform::IsHeadless()` (keyed off MOZ_HEADLESS); otherwise it calls the platform `nsIWidget::CreateTopLevelWindow()` which the headless widget backend never defines → abort stub.
- Register a no-op `nsBaseAppShell` subclass under `NS_APPSHELL_CID` — the headless widget toolkit registers NO appshell, so `nsAppStartup::Init()`'s `do_GetService(NS_APPSHELL_CID)` fails and `@mozilla.org/toolkit/app-startup;1` (Services.startup) is FACTORY_NOT_REGISTERED. We pump via SpinEventLoopUntil so the appshell can be inert.
- GRE/APP dir split: `NS_InitXPCOM` binDir = `/gre/browser` (makes resource:/// = front-end), plus a directory provider supplying `NS_GRE_DIR`=/gre AND `nsIDirectoryServiceProvider2::GetFiles(NS_APP_PREFS_DEFAULTS_DIR_LIST)` = /gre/browser/defaults/preferences (no omnijar → firefox.js default prefs only load via that dir-list branch).
- Stage fonts at `/gre/browser/fonts` too (FindFonts scans the app dir).
- `StartupTimeline::Record(PROCESS_CREATION/START/MAIN)` — XRE records these; tabs.js reads `getStartupInfo().start` and throws without them.
- Tabs must be non-remote (single process, no content process): force `browser.tabs.remote.autostart`/`fission.autostart`=false and open sites with `gBrowser.addTab(url,{forceNotRemote:true})`. The initial tab browser is a broken half-remote state (remote=true, remoteType=null), so open a FRESH tab. Drive it by evaluating JS in the chrome global (AutoEntryScript + JS::Evaluate), deferred via `setTimeout(...,0)` because addTab's heavy synchronous setup hangs if run nested inside the C++ command handler.

WebExtensions (WORKING): install + run, verified by embed-chrome/ext-test.cjs (a temp MV2 content-script extension paints example.com magenta + injects a banner). Fullscreen + live resize (DONE): embed-chrome/index.html is a bare fullscreen canvas (no header/buttons/urlbar/log; dark loading overlay until browser.xhtml renders). The engine renders the chrome at the LIVE window size and reflows on resize. C++: EnsureSize(w,h) (embed-xul.cpp) resizes the AppWindow (chrome) / docshell base window (content) via SetPositionAndSize only when changed; called from xul_paint/gpu_present/EnsureBrowser. Verified resize-test.cjs (1100x700/1600x1000/820x560: chrome reflows, not scaled).

Popups (menus, context menus, app-menu) — FIXED (verified: right-click URL bar shows the real text context menu rendered on canvas; rclick-test.cjs). All embedder-side, libxul source UNCHANGED. Two compounding root causes from bypassing the per-toolkit nsAppShell: (1) the gfx ScreenManager was empty (real FF installs HeadlessScreenHelper via nsAppShell; our no-op EmbedAppShell didn't), so nsMenuPopupFrame::GetConstraintRect saw a 0-size screen and clamped every popup's mUsedRect to 0x0; (2) the headless popup widget never sizes the popup frame, so DoReflow gave the popup reflow root 0 available inline-size and content collapsed. Fix: RefreshScreen(w,h) in embed-xul.cpp calls ScreenManager::Refresh() with one Screen == window size (at EnsureBrowser + EnsureSize); xul_paint's popup loop, when CalcWidgetBounds==0, SetSize(window)+FrameNeedsReflow+flush+SetPopupPosition so it reflows to content (self-limiting); the loop composites visible popups (GetVisiblePopups -> nsLayoutUtils::PaintFrame at CalcWidgetBounds) since popups live in their own widgets RenderDocument skips; do_mouse evType=3 -> "contextmenu" + the canvas contextmenu handler sends it (synthesized right mousedown/up alone doesn't generate eContextMenu headless). DEBUGGING LESSON: menu-test.cjs's CDP capture filtered console lines lacking "popup", which made my libxul printf diagnostics look like LayoutPopup was never called (red herring); always capture unfiltered when diagnosing. geckoEval(js) hook (op=5 -> RunChromeScript) drives chrome JS from tests.

Broken initial/default tab (FIXED 2026-06-17): the first tab the chrome opens at startup was stuck in a broken state (couldn't close/navigate; interacting "slowly corrupted memory"); subsequent tabs were fine. ROOT CAUSE: tabbrowser.js `_setupInitialBrowserAndTab` picks the initial browser's remoteType, and its fallback branch (taken when `gBrowserInit.uriToLoadPromise` is an unresolved PROMISE, not a string -- which is our case: the window is opened via CreateTopLevelWindow with NO url arg, so the URI waits on SessionStore) HARDCODES `E10SUtils.PRIVILEGEDABOUT_REMOTE_TYPE` -- the ONLY remoteType branch that ignores `gMultiProcessBrowser`. So even with e10s off the initial browser is created REMOTE; single-process has no content process -> stuck `remote=true / remoteType=null`, and every interaction pokes a process-less frameLoader. addTab/"+" tabs use a path that respects gMultiProcessBrowser=false -> non-remote -> fine. FIX (front-end JS, source + gre-stage copy, relink no libxul): gate that fallback -> `remoteType = gMultiProcessBrowser ? PRIVILEGEDABOUT_REMOTE_TYPE : E10SUtils.NOT_REMOTE;` in browser/components/tabbrowser/content/tabbrowser.js. Verified: initial New Tab renders + is interactable.

Popup compositing use-after-free (FIXED 2026-06-17): the embedder's `composite_visible_popups` (embed-xul.cpp; called by xul_paint AND paint_popup_overlay) held raw `nsMenuPopupFrame*` from GetVisiblePopups ACROSS `FlushPendingNotifications(FlushType::Layout)` (the 0-sized-popup reflow/reposition dance). A layout flush can reflow + run script that DESTROYS popup frames, so the post-flush `pf->SetPopupPosition`/`CalcWidgetBounds`/`PaintFrame(ctx,pf,...)` wrote through freed memory (and later `popups[]` entries dangled too) -> intermittent memory corruption whenever a popup was showing (i.e. on interaction). FIX: never hold a raw popup pointer across a flush -- size all 0-sized popups, flush ONCE, RE-QUERY GetVisiblePopups, reposition, flush, re-query, then a FLUSH-FREE paint pass. Embedder-only, relink. (Was wrongly attributed to the broken initial tab; it's a separate latent bug.)

Limitation: the tab's content BrowsingContext is NOT reachable from the chrome docshell/BC tree in C++ (lazy frame-loader presentation), so xul_chrome_load_content can't observe load completion — it pumps the event loop a bounded ~400k iterations and relies on the continuous paint loop to render the result.
