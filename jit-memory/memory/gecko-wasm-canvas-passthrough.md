*** RESOLVED (2026-06-15) — GPU-MODE PASSTHROUGH WORKS. *** Content WebGL (canvas.getContext('webgl2')) renders on the real host GPU via a WORKER-LOCAL OffscreenCanvas context and composites onto the page in GPU mode (?gpu=1&glpass=1). Verified: webgl-test.html gradient triangle renders on screen, 0 GL errors, no multi-context conflict. Flag off by default = no regression (velzie.rip/x.com still clean).

THE REAL BLOCKER WAS A SILENT BUILD-STALENESS BUG, not the architecture: GLContextProviderEmscripten.cpp used `EM_ASM_INT({... var attrs={majorVersion:2, minorVersion:0, ...} ...})`. The C preprocessor splits macro args on commas (only PARENS protect them, NOT braces), so the JS object-literal commas split the EM_ASM_INT macro -> the JS parsed as C++ -> `error: unknown type name 'var'`. The Unified_cpp_gfx_gl0.o compile FAILED, BUT `mach build binaries` still returned rc=0, so builds silently shipped the OLD GL provider. Every GPU-passthrough test for hours ran stale code -> false "multi-context conflict" conclusions. FIX: build attrs with `attrs.x = y;` assignments (no top-level commas). Force the recompile after a silent failure: `rm obj-full-emscripten/gfx/gl/Unified_cpp_gfx_gl*.o obj-full-emscripten/dist/bin/libxul.so` then mach build. VERIFY code landed: `grep -ac <marker> obj-full-emscripten/dist/bin/libxul.so`.

WHY WORKER-LOCAL OFFSCREENCANVAS WORKS: emscripten's GLctx + GL.contexts + handle tables are PER-THREAD JS state. The compositor's context is proxyContextToMainThread=ALWAYS (thread 0). The conflict was content GL touching thread 0's GLctx. Fix: GLContextProviderEmscripten::CreateContent does `EM_ASM_INT` on the CONTENT thread: `new OffscreenCanvas(...).getContext('webgl2',attrs)` + `GL.registerContext(glctx,attrs)` -> a context OWNED BY THE CONTENT THREAD (GL.registerContext stamps _pthread_self()). emscripten_webgl_make_context_current then succeeds LOCALLY on the content thread (verified result=0, same thread as CreateContent), so content GL runs on the content thread's own GLctx, never touches thread 0 -> no conflict, no proxy tax. Reuses emscripten's full GL impl (NO 150-fn reimplementation needed). SurfaceFactory_Basic glReadPixels readback (auto fallback, WebGLContext.cpp:1210) feeds WebRender, which composites the canvas.

FLAG must be captured on the APP thread: the provider runs on a worker where getenv() may not see ENV. embed-xul.cpp main(): `int g_gl_passthrough = getenv("GECKO_GL_PASSTHROUGH")?1:0;` + `extern "C" int gecko_gl_passthrough_enabled(){return g_gl_passthrough;}`; the provider reads it via a weak extern (like wisp_wakeword). index.html ?glpass=1 -> ENV.GECKO_GL_PASSTHROUGH.

SHADER FIX (still needed): ANGLE hashes identifiers to `webgl_<hash>`, reserved on the host WebGL -> changed kHashedNamePrefix to "mzwgl_" under #if __EMSCRIPTEN__ in gfx/angle/checkout/src/compiler/translator/HashNames.cpp.

STILL OPEN (minor): (1) readback B/R swap -- content clearColor dark-blue composites as dark-red (R/B swapped), same channel-order issue as raster images (WebGL2 lacks texture swizzle/BGRA); triangle vertex colors look right. (2) SOFTWARE-mode (no GECKO_GPU) display: content WebGL renders+reads-back (WEBGL_OK) but the canvas shows BLANK -- the sync PresShell::RenderDocument snapshot doesn't capture the async WebGL canvas surface; GPU mode (WebRender) composites it fine. (3) heavy real WebGL sites untested (WISP page-load flakiness; data: document scripts may be blocked).

The WebGL canvas composites through WebRender's ASYNC CanvasRenderer (WebRenderCanvasRenderer), which the sync RenderDocument snapshot doesn't capture. 2. GPU embedder paint (gpu_present -> WebRender, GECKO_GPU=1): WebRender WOULD composite the canvas natively, BUT there's a MULTI-CONTEXT conflict -- content GL ends up corrupting the compositor's thread-0 WebGL context -> "glDrawElementsInstanced: Must have element array buffer bound" on the compositor -> ALL compositor draws skipped -> entire page blank ("0 px"). ATTEMPTED FIX (didn't work): register the content context WORKER-LOCAL via `new OffscreenCanvas().getContext('webgl2')` + `GL.registerContext` on the content thread (GLContextProviderEmscripten::CreateContent, current code), so it'd use that thread's own GLctx. STILL CONFLICTS in GPU mode -- content GL still reaches thread 0. Mechanism notes from the generated gecko.js: `_glXxx` are bare (`GLctx.drawArrays(...)`, NO per-call proxy) and use the calling thread's `GLctx`; `_emscripten_webgl_make_context_current_calling_thread` makes a context current only if it's in THIS thread's GL.contexts (else proxies). So if content GL truly ran on the content worker with its local OffscreenCanvas context, it would NOT touch thread 0 -- the persistent conflict implies Gecko executes the content GL (HostWebGLContext / readback) on a thread that shares thread 0's GLctx, OR make-current for the registered context isn't taking locally. NOT yet root-caused. THE ROBUST FIX per velzie: a DIRECT JS GL forwarder for content -- the content gl::GLContext's symbol loader returns custom functions that call the content OffscreenCanvas context's methods DIRECTLY (own handle tables), never touching emscripten's global GLctx/proxying. That's a ~150-function reimplementation of library_webgl.js against a dedicated context (large but bounded). Investigate where Gecko runs content GL (webgl.out-of-process / HostWebGLContext thread) first -- if it's already the content worker, the OffscreenCanvas approach may just need the make-current/symbol-loader wired to bypass the proxying stubs.

## WebGL1 content fix (2026-06-16) — honor the requested WebGL version
glpass content contexts (GLContextProviderEmscripten::CreateContent) hardcoded
majorVersion=2 + getContext("webgl2"), forcing WebGL2 on ALL content. This broke real
WebGL1 sites: e.g. velzie.rip's pink-noise background shader does `#extension
GL_EXT_frag_depth : require` (+ GL_EXT_shader_texture_lod), WebGL1-only extensions that
are CORE (and thus NOT exposable extensions) in WebGL2 -> the require errors -> shader
won't compile -> blank/no GL. FIX: CreateContent picks the version from the requested
flags -- `int wantWebgl2 = bool(desc.flags & CreateContextFlags::PREFER_ES3)` (WebGLContext
sets PREFER_ES3 for WebGL2, leaves it off for WebGL1) -> getContext($0?"webgl2":"webgl") +
majorVersion=$0?2:1. ALSO build-embed-full.sh: -sMIN_WEBGL_VERSION=1 (was 2) so emscripten's
GL layer supports WebGL1 contexts (MAX stays 2; the WebRender compositor still uses WebGL2).
Verified: velzie.rip loads with ?gpu=1&glpass=1, 0 shader errors, the pink-noise GL
background renders (screenshot: page content over visible pink/magenta noise). This was
exposed while validating the OffscreenCanvas compositor work (see [[gecko-wasm-gpu-integration]]).
