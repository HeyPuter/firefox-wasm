Goal (2026-06-14/15): real GPU acceleration — in-process WebRender composites directly to the page `<canvas>` via WebGL2, single-process (no GPU process). Builds on [[gecko-wasm-webgl-canvas]] (de-risk facts) and [[gecko-wasm-runtime-frontier]].

ARCHITECTURE: windowless browser (PuppetWidget) -> in-process WebRender compositor (layers.gpu-process.enabled=false) -> RenderThread -> RenderCompositorOGL -> GLContextProviderEmscripten (WebGL2, proxied to main browser thread) -> page canvas #screen. GPU is OPT-IN via env GECKO_GPU=1 (index.html?gpu=1 sets it), keeping the software RenderDocument+blit path the default.

ROOT-CAUSE LANDMINE (emscripten): mmap(fd) does NOT read/share the fd's content -- it returns private zero-filled anonymous memory. This broke TWO things, both fixed:
- gfx/2d/UnscaledFontFreeType.cpp GetFontFileData: used mmap to read the font file -> all zeros. Fixed: read() the file (emscripten read() works on MEMFS). Also GetFontDescriptor returns false for emscripten so WebRender's glyph rasterizer uses RAW font bytes (FT_New_Memory_Face) not a native path (CString::new(path) crash + worker file I/O).
- ipc/glue/SharedMemoryPlatform_posix.cpp Platform::Map: used mmap(shm_fd, MAP_SHARED) -> zeros, so ALL IPC SharedMemory (WebRender resource updates: fonts/images/blobs, AND SharedPrefMap) transfers zeros. Fix IN PROGRESS: back shmem with heap buffers keyed by fd inode (wasm heap is shared across pthreads via SAB), Unmap is a no-op (leak). CAUTION: broke startup (SharedPrefMap MOZ_RELEASE_ASSERT(map.IsValid())) -- the aFixedAddress bail made Map return Nothing; removed it. Watch freeze: SharedPrefMap freezes then maps a read-only fd; if the frozen fd has a different inode than the write fd, may need to alias inodes in Platform::Freeze.

DONE: TEXT RENDERS on the GPU ("Hello GPU rendering 123" + headings/paragraphs/bold + gradients + solid boxes all composite). The IPC SharedMemory fix (file-header-pointer live sharing, ipc/glue/SharedMemoryPlatform_posix.cpp Platform::Map/Unmap under #if __EMSCRIPTEN__) was the key: emscripten mmap(fd) and fstat(fd) are both broken for shm fds, but read/pread/pwrite work and the wasm heap is shared across pthreads, so the object's data lives in ONE heap buffer whose 32-bit pointer is stored in an 8-byte header (magic "MOZS" + ptr) at the start of the shm file; first Map allocates+writes the header, every other Map (any fd/thread) reads the pointer back -> same live buffer. Unmap is a no-op (leak; WebRender recycles). This unblocked fonts AND SharedPrefMap AND any shmem-backed resource (images etc.). NOTE: earlier inode-keying failed (fstat errno 44 on shm fds) and flush-on-unmap failed (WebRender writer doesn't unmap before reader reads) -- the header-pointer live-share is what works.

CHROME BUILD GPU (2026-06-15, FIXED): the full-chrome build (embed-chrome, GECKO_CHROME=1) rendered an all-white page under ?gpu=1 with `RenderCompositorSWGL failed mapping default framebuffer, no dt` — it composited via SOFTWARE WebRender (RenderCompositorSWGL), unlike the windowless embed-xul build which uses hardware RenderCompositorOGL. Root cause: the chrome build sets `MOZ_HEADLESS=1` (embed-xul.cpp:235, chrome-only, needed for AppWindow->CreateHeadlessWidget), so `gfxPlatform::IsHeadless()` is TRUE; `gfxPlatform::InitCompositorAccelerationPrefs()` does `if (IsHeadless()) feature.ForceDisable(...)` on Feature::HW_COMPOSITING, and that ForceDisable OVERRIDES `layers.acceleration.force-enabled` ("safe/headless/record modes override everything"), so no pref can flip it. With HW_COMPOSITING off, gfxConfigManager::ConfigureWebRender force-disables Feature::WEBRENDER (UnavailableNoHwCompositing) -> hasHardware=false -> gfxVars::SetUseSoftwareWebRender(true) -> RenderCompositor::Create picks SWGL (which needs a window DrawTarget the headless widget can't supply -> "no dt" -> white). The windowless build never hits this (IsHeadless()=false). FIX: gfx/thebes/gfxPlatform.cpp wraps that headless HW_COMPOSITING ForceDisable in `#ifndef __EMSCRIPTEN__` — on emscripten "headless" only means no native window, GLContextProviderEmscripten still backs the compositor with a real WebGL2 context, so HW_COMPOSITING (and WebRender hardware) must stay enabled. No widget/GL-context change needed: RenderThread.cpp CreateGLContext's emscripten branch passes nullptr for the compositor widget (widget-independent) and keys hardware on !UseSoftwareWebRender(); gpu_present() gets the widget generically via nsContentUtils::GetWidget(presshell) so it already drives the chrome window's HeadlessWidget compositor. After the fix the chrome window's `windowUtils.layerManagerType` is "WebRender" (was "WebRender (Software)") and content composites on the GPU. libxul change -> mach build binaries + restrip-relink-web.sh. Test: embed-chrome/gpu-chrome-test.cjs (asserts layerManagerType + no SWGL/GFX1- errors; screenshot gpu-chrome.png).

GPU POPUP COMPOSITING (2026-06-15, FIXED): once the chrome composited on the GPU, opening any popup (context menu, app menu, <select>) made it flicker at the canvas BOTTOM-LEFT, alternating with the main view and white. Cause: a XUL popup is a SEPARATE top-level widget and a SEPARATE display root (NS_FRAME_IN_POPUP -> nsLayoutUtils::GetDisplayRootFrame stops at the popup frame, so popups are NEVER in the main window's WebRender scene -- making them widgetless does NOT help). With HW_COMPOSITING now on, each popup widget got its OWN WebRender compositor (nsIWidget::ShouldUseOffMainThreadCompositing is a global, true for all widgets), and via nsXULPopupManager::PaintPopups -> PaintAndRequestComposite each popup's compositor presented its small framebuffer to the SAME single page <canvas>, clearing it and landing at the GL origin (bottom-left) -- multiple compositors fighting one canvas. Content has no separate widget (in-process <browser> subdocs paint into the chrome window's scene), which is why page loads were fine; only popups broke. FIX (single-canvas embedding can't do in-Gecko multi-window GPU compositing, so keep popups off the GPU and overlay them): (1) libxul widget/headless/HeadlessWidget.cpp+.h: override ShouldUseOffMainThreadCompositing() to return false for `mWindowType==WindowType::Popup` (#if __EMSCRIPTEN__) -> popup widgets use the FALLBACK renderer, so PaintPopups' `renderer->AsFallback()` branch just Invalidates (never composites to the canvas) -> main scene stays stable on the GPU. (2) embedder embed-xul.cpp: factored the working software popup loop into `composite_visible_popups(gfxContext*,PresShell*,w,h,appPerCss)` (used by xul_paint), and added `paint_popup_overlay(w,h)` that paints the visible popups into a transparent canvas-sized BGRA buffer (reused static), returned via the otherwise-unused result/resultLen in GPU mode (null when no popup is open). (3) JS embed-chrome/index.html: a 2D `#popups` overlay canvas stacked over the WebGL `#screen` (z-index 5, pointer-events:none); `drawPopupOverlay` un-premultiplies the BGRA + putImageData (preserves rounded/AA alpha so the GPU scene shows through), clears when empty. So content+chrome stay GPU-accelerated; popups (small, transient, simple) are software-painted onto the overlay. Verified: embed-chrome/gpu-popup-test.cjs (context menu renders at the right position over the GPU scene, no SWGL errors, dismiss clears it; screenshots gpu-popup-open/closed.png).
