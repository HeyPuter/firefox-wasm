GOAL (2026-06-15, user): "pass guest webassembly to host webassembly. exports should go through." Our wasm32-emscripten SpiderMonkey has no in-process wasm compiler (JIT disabled), so the `WebAssembly` global was NEVER exposed to content (baseline test: `WASM_FAIL WebAssembly is not defined`). This feature exposes it and routes guest wasm to the HOST browser's WebAssembly engine (real JIT), bridging exports back.

WORKS (verified, ALL entry points): embed-xul/wasm-stream-test.html -> ALL_PASS for instantiate(bytes), instantiateStreaming(fetch), compileStreaming+new Instance, compile+instantiate(module), new Module+new Instance. embed-xul/wasm-import-test.html -> WASM_IMPORT_OK: a module importing env.dbl(i32)->i32 and exporting calc(x)=dbl(x)+1 returns calc(5)=51 (import arg in, import return value flows back into the host wasm, final result out -- full bidirectional bridge). embed-xul/wasm-test.html -> add(40,2)=42 + 2M-call loop in ms=592 (~296ns/call = the per-call boundary tax: guest JS -> SpiderMonkey native -> js-library -> host wasm; host module is JIT-compiled). Great for compute-heavy/few-call wasm; boundary cost dominates for chatty wasm.

ENTRY POINTS ROUTED (all under `if (wasm::UseHostPassthrough())`, #if __EMSCRIPTEN__): WebAssembly_instantiate (bytes->{module,instance} OR host-module->instance), WebAssembly_compile (->host module obj), WebAssembly_instantiateStreaming + WebAssembly_compileStreaming (Response->arrayBuffer()->host, via a promise chain), WasmModuleObject::construct (new Module), WasmInstanceObject::construct (new Instance). Helpers (forward-declared before WasmModuleObject::construct since they're used at lines 1919/2435 but defined near WebAssembly_instantiate): HostCompileBytes, HostMakeModuleObject/HostModuleHandle (opaque module = plain obj w/ non-enum `__wasmHostHandle`), HostBindAndInstantiate (import loop + wasmhost_instantiate), HostBuildInstanceObject, HostPassthroughInstantiate, HostStreamingInstantiate (+ HostStream_On{Response,Bytes,Reject} native promise reactions; closure = plain obj {promise,importObj,instantiate} in the callback's extended slot 0).

IMPORTS (host->guest callbacks) WORK: two-phase -- wasmhost_compile(bytes)->handle (host Module + import/export lists); C++ enumerates imports, finds each fn in importObj, RegisterImportCallback (stores it in a leaked PersistentRootedObject indexed map, returns id); wasmhost_instantiate(handle, idsPtr, count) builds hostImports where each function import = a host shim `(...a)=>Module._wasmhost_invoke_import(id, scratch, argc)`. The exported `wasmhost_invoke_import(id,argsPtr,argc)` (EMSCRIPTEN_KEEPALIVE + in EXPORTED_FUNCTIONS) retrieves cx via js::TlsContext.get(), reads args into a local (reentrancy-safe), JS_CallFunctionValue the stored guest fn, returns the result as double. Non-function imports (memory/table/global) are left unbound -> a module that REQUIRES them throws at instantiate (caught -> rejects).

THE GATE: js/src/vm/GlobalObject.cpp:177 skips JSProto_WebAssembly when `!wasm::HasSupport(cx)`, and CreateWebAssemblyObject (WasmJS.cpp ~6020) `MOZ_RELEASE_ASSERT(HasSupport(cx))`. HasSupport -> HasPlatformSupport() -> false at `!HasJitBackend()`. FIX (js/src/wasm/WasmFeatures.cpp): added `bool wasm::UseHostPassthrough(){return true;}` (#if __EMSCRIPTEN__) + `HasSupport` returns true early when UseHostPassthrough. That alone installs WebAssembly AND satisfies the assert (no GlobalObject.cpp change needed). Decl in WasmFeatures.h.

THE BRIDGE (avoids the EM_ASM-comma trap by being a plain js-library file, not EM_ASM): embed-xul/wasm-host-bridge.js, wired via `--js-library` in build-embed-full.sh. Functions (called from C++ as extern "C"): wasmhost_instantiate(ptr,len)->handle (does `new WebAssembly.Module(bytes); new WebAssembly.Instance(mod,{})` on the host, keeps a per-worker registry on `globalThis.__whReg`, returns int handle or -1); wasmhost_export_count/_kind/_name (metadata; _name uses $stringToUTF8/$lengthBytesUTF8 deps); wasmhost_call(h,idx,argsPtr,argc)->double (reads argc f64 from HEAPF64, calls export idx, coerces result to number). Registry is per-worker on globalThis -- works because content JS for one global is single-threaded (instantiate + every export call share the Gecko-main-thread worker).

THE C++ (js/src/wasm/WasmJS.cpp, #if __EMSCRIPTEN__): extern decls for the 5 wasmhost_* fns; `WasmHostExportCall` (JSNative: reads host handle from extended slot 0, export idx from slot 1; ToNumber each arg into a double buf[64]; wasmhost_call; rval.setNumber). `HostPassthroughInstantiate(cx, bytesObj, importObj, out)`: IsBufferSource+CheckedUnwrapStatic to get the guest ptr/len (same as GetBytecodeSource), wasmhost_instantiate, then for each export build the exports object -- function exports become `NewNativeFunction(WasmHostExportCall, 0, nameAtom, gc::AllocKind::FUNCTION_EXTENDED)` with setExtendedSlot(0,handle)/(1,idx); other kinds -> null. Builds {module:{}, instance:{exports}} (plain objects, NOT WasmInstanceObject -- `instanceof WebAssembly.Instance` is false, acceptable). Hooked into WebAssembly_instantiate right after GetInstantiateArgs: `if (wasm::UseHostPassthrough()) { HostPassthroughInstantiate; PromiseObject::resolve; return; }`.

ALWAYS-ON (no flag): content wasm was fully broken before, so enabling is strictly better for wasm-using sites. RISK to weigh: a site that feature-detects `typeof WebAssembly!=='undefined'` now gets true and will TRY wasm; modules WITH IMPORTS currently fail at instantiate (we pass `{}`), so a site with a non-wasm fallback could regress. If that bites, gate UseHostPassthrough behind a flag (capture on the app thread like GECKO_GL_PASSTHROUGH -- getenv is unreliable on worker threads).

MEMORY (copy bridge -- WORKS, verified wasm-mem-test.html MEM_PASS): the host wasm's linear memory is a host ArrayBuffer the guest can't alias, so content gets a "mirror" -- `JS::NewArrayBufferWithUserOwnedContents` over a js_pod_calloc'd guest-heap region (stable ptr, never GC-moved), wrapped as `{__wasmHostObjId, buffer}`. The mirror is the SAME storage content's cached `HEAPU8 = new Uint8Array(mem.buffer)` views, so writes land where content reads. SYNC happens at the boundaries where host<->guest memory changes hands: in wasmhost_call (guest->host before the export runs, host->guest after) AND in the import shim (host->guest before invoking the guest callback, guest->host after) -- because emscripten's imported syscalls read/write the heap mid-call. Only the CURRENT (live) size is copied, not a max. js-library: __whObj[] (host Memory/Table/Global), __whObjMirror{objId:{ptr,len}}, $whSyncMem, wasmhost_mem_new/table_new/global_new/mem_bytelength/obj_set_mirror/export_register_mem. C++: HostObjId, HostMakeMemoryWrapper (mirror+AB+wrapper, registers mirror with host), HostMakeObjIdWrapper (table/global, id only), routed WasmMemory/Table/Global Object::construct (parse desc, wasmhost_*_new, wrap), HostBindAndInstantiate binds memory/table/global imports by objId, HostBuildInstanceObject wraps memory exports. reg[handle].memObjId tracks the instance's memory for sync.

FIXED the localhost:5000 crash: `new WebAssembly.Memory()` hit native WasmMemoryObject::construct -> CreateWasmBuffer -> `RuntimeError: unreachable` (no in-process wasm memory). Now routed. localhost:5000 IS the gecko-wasm frontend itself (serves gecko.js) -> loading it = gecko-in-gecko; the inner instance now gets past memory creation to "downloading data / loading wasm" instead of crashing.

WHY NOT byte-granular copy (user asked): can't intercept individual reads on a real TypedArray -- emscripten caches one `new Uint8Array(mem.buffer)` and reads arbitrary offsets with no hook, and `new Uint8Array(realBuffer)` bypasses any Proxy. So we copy the live size at the hand-off boundaries (export + import calls), not per-byte.

SHARED MEMORY (emscripten pthreads) -- "bad memory" fix (verified wasm-shmem-test.html SHMEM_PASS): a pthread module does `new WebAssembly.Memory({shared:true})` then asserts `wasmMemory.buffer instanceof SharedArrayBuffer` (throws "bad memory" otherwise). So for a shared memory the mirror must be a real SAB, not the user-owned ArrayBuffer. HostMakeMemoryWrapper(cx, objId, shared): if shared, mirror = `JS::NewSharedArrayBuffer(cx, len)` (engine-owned, non-movable so the ptr is stable) and the sync ptr comes from `JS::GetSharedArrayBufferData`; else the user-owned AB. `shared` comes from the Memory ctor desc, or from `wasmhost_mem_is_shared(objId)` (`buffer instanceof SharedArrayBuffer`) for exported memory. Needs `#include "js/SharedArrayBuffer.h"`. NewSharedArrayBuffer needs the realm's getSharedMemoryAndAtomicsEnabled (the javascript.options.shared_memory pref, on) -- independent of COI. *** THREADING CAVEAT: this only fixes the single-threaded path. If the module actually spawns pthreads (content Workers) and expects them to TRULY share the memory, it WON'T work: __whObj/__whReg are per-emscripten-worker, so each content worker's host wasm instance gets its OWN host memory -- the host instances don't share, and the per-thread copy-at-boundary sync diverges. True shared-memory threading needs all content workers' host instances to share ONE host SAB + a cross-worker registry (big follow-up). ***

OPEN (remaining frontier):
1. memory.grow: the mirror is allocated at the memory's CURRENT byteLength at wrap time; a host-driven grow (or content calling memory.grow) past that isn't reflected -- the extra pages aren't synced and the mirror AB stays the old size. Need: detect grow (host bytelength change) after a call, reallocate the mirror + a new user-owned AB, update wrapper.buffer (content re-derives HEAPU8 on grow). Big for emscripten with ALLOW_MEMORY_GROWTH.
2. Perf: full live-size copy per export AND per import call. Brutal for large heaps + chatty modules (e.g. the inner gecko's hundreds of MB). Optimize later (dirty-range tracking is hard without write interception; could cap, or only sync if an export/import actually touches memory).
3. Table/Global: constructors create host objects + bind as imports, but JS-side element (table.get/set/grow) and value (global.value) access is NOT bridged. Table funcref elements that cross to JS aren't wrapped.
4. i64/BigInt + reference-type args/results: coerced through double (lossy for i64).
5. Multiple instances of one Module clobber reg[handle].inst; exception propagation from an import callback is swallowed; validate() + Module reflection unhandled.
